## 1. 목표
- 하네스 v1 위에서 첫 실전 프로젝트를 안정적으로 투입한다.
- 첫 프로젝트는 `contest-mvp` 기반 **API 활용 대회 준비**다.
- 현재 1차 목표는 **구현 착수 전, 현재 상황을 명확히 브리핑 가능한 상태**를 만드는 것이다.
- 제품 방향의 1순위 가설은 **중소규모 현장용 위험성평가·TBM 코파일럿**이다.
- 핵심 흐름은 **기상/현장 위험 신호 → 법령·KOSHA 근거 → 현장조치 → 위험성평가/TBM 산출물**이다.

## 3. 핵심 리스크
1. Codex는 현재 봇간 자동대화보다 **사람 트리거형 운영**이 더 안정적이다.
2. `contest-mvp` 웹앱 본체에는 아직 **live adapter가 본격 이식되지 않은 상태**다.
3. 라이브 API 일부는 `429` 또는 timeout이 있어 **재시도/backoff 보강**이 필요하다.
4. 제품 메시지는 강하지만, **데모 화면/흐름 고정**은 아직 덜 끝났다.
5. Telegram privacy mode가 켜져 있으면 비멘션 그룹 메시지를 일부 놓칠 수 있다.

## 4. 즉시 다음 액션
1. 이 문서를 기준으로 현재 상태를 공유한다.
2. 대회용 제품 스토리를 **위험성평가·TBM 코파일럿**으로 확정한다.
3. 첫 데모 질문 / 화면 흐름 / 시연 시나리오를 고정한다.
4. E2E 프로토타입의 핵심 흐름을 `contest-mvp` 웹앱에 이식할 우선순위를 정한다.
5. `429`/timeout 대응용 retry/backoff 및 fallback 정책(mock/live/API 실패 시 응답)을 정리한다.
