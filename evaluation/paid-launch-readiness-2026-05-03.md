# SafeClaw 유료상품 런칭 가능성 검토

작성일: 2026-05-03

## 결론

현재 SafeClaw는 `제출·시연 가능한 제품`과 `파일럿 유료상품` 사이에 있습니다. 핵심 엔진은 충분히 강합니다. 한 줄 입력, 공공 API 조합, 11종 문서팩, 다국어 공지, 메일·문자 전파, Supabase 저장, 지식 DB, PDF/HWPX/XLS 출력까지 연결된 증거가 있습니다.

다만 지금 그대로 `셀프서브 유료 SaaS`로 열기에는 아직 닫히지 않은 운영 공백이 있습니다. 추천 런칭 기준은 공개 결제가 아니라 `관리자 동행형 유료 파일럿`입니다. 즉, 사용자를 선별해 온보딩하고, 첫 현장 서식 검수와 전파 채널 설정을 같이 해주는 방식이면 유료로 닫을 수 있습니다.

## 런칭 판정

| 구분 | 판정 | 이유 |
|---|---|---|
| 공모전 제출 | 가능 | live API 조합, 문서팩 생성, 저장, 전파, 다운로드 증거가 있음 |
| IR/데모 | 가능 | v2 화면, `/demo`, `/why`, `/trust`, `/roadmap` 흐름이 갖춰짐 |
| 관리자 동행형 유료 파일럿 | 조건부 가능 | 고객별 온보딩과 서식 검수를 붙이면 가치가 설명됨 |
| 무인 셀프서브 유료 SaaS | 아직 보류 | 저장/이력, 전파 실패 대응, 문서 검수, 화면 일관성의 운영 리스크가 남음 |
| 공공기관·대기업 표준 제출 자동화 | 아직 보류 | 공식 제출을 대체하는 수준이 아니라 공식자료 기반 초안과 검토 보조 수준 |

## 강점

- `작업 입력 -> API 근거 -> 문서팩 -> 편집 -> 다운로드 -> 전파 -> 저장`의 핵심 루프가 실제 코드로 연결되어 있습니다.
- 제출 스모크 기준으로 API 조합 문서 생성, 관리자 저장, 서식 다운로드는 통과했습니다.
- 메일·문자 전파는 실제 수신 증빙이 있고, n8n 응답과 전파 로그 구조도 있습니다.
- 지식 DB smoke가 통과했고, KOSHA/법령/위험요인 매칭을 문서 보완으로 연결하는 기반이 있습니다.
- 위험성평가표, 작업계획서, 안전작업허가, TBM 브리핑의 서식 깊이가 최근 보강되어 단순 메모형 문서에서 벗어나고 있습니다.

## 유료상품 전환 전 남은 차단 리스크

### 1. 전파 채널 신뢰

메일과 문자는 실제 수신 확인이 있으나, 자동 smoke에서는 SMS provider 응답이 실패로 남은 기록이 있습니다. 유료상품에서는 “보냈다”와 “도착했다”를 분리해 보여줘야 합니다.

필요 조치:
- 전파 결과를 `접수`, `전달`, `실패`, `확인 필요`로 분리합니다.
- 문자 실패 시 provider 원문과 재시도 안내를 관리자 화면에 남깁니다.
- 카카오·밴드는 승인 전까지 명확히 잠급니다.

### 2. 저장·이력의 제품 신뢰

Supabase 저장은 통과했지만, 전체 화면에서 사용자가 “내 문서가 안전하게 저장됐다”고 확신하는 경험은 아직 약합니다.

필요 조치:
- 저장 완료 후 문서팩 ID, 저장 시각, 저장 항목을 모든 주요 화면에서 일관되게 표시합니다.
- `/archive`에서 실제 저장 목록, 전파 로그, 교육기록이 하나의 작업 이력으로 보이게 합니다.
- 비회원 임시 저장과 관리자 저장의 차이를 더 선명하게 표시합니다.

### 3. 서식 품질의 일관성

위험성평가표와 TBM은 많이 올라왔지만, 모든 문서가 동일한 표 형태로 보이는 순간 “자동 생성 문서” 느낌이 다시 납니다. 유료 전환 기준에서는 문서별 목적이 눈에 띄어야 합니다.

필요 조치:
- 위험성평가표는 평가 행 중심, 작업계획서는 순서·장비·인원·작업계획도 중심, TBM은 참석·복창·조치 확인 중심, 허가서는 승인·첨부·작업조건 중심으로 확실히 갈라야 합니다.
- PDF/HWPX/XLS는 사람이 열어본 현장 검수 로그를 남겨야 합니다.
- “공식자료 기반 초안이며 현장 검토 후 사용” 문구를 모든 제출형 출력에 포함합니다.

### 4. 화면 IA와 디자인 일관성

디자이너 프로토타입 기반 셸은 일부 반영됐지만, 내부 기능 화면은 아직 “기존 기능을 새 껍데기에 얹은 상태”가 섞여 있습니다.

필요 조치:
- 운영 메뉴에서 프로토타입 번호, A/B안, 화면 수 노출을 제거합니다.
- 실제 메뉴는 `제품`, `작업공간`, `문서팩`, `근거`, `작업자`, `전파`, `이력`, `지식 DB`, `설정` 정도로 고정합니다.
- 핵심 업무 화면은 디자인 시안의 구조를 가져오되, 장식보다 기능 조작성을 우선합니다.

### 5. 지식 DB의 사용자 체감

지식 DB smoke는 통과하지만, 사용자는 아직 “이 근거가 왜 내 문서에 들어갔는지”를 한 번에 이해하기 어렵습니다.

필요 조치:
- 근거 카드를 `문서 반영 위치`, `반영 문장`, `출처`, `검토 필요 여부`로 재정렬합니다.
- 보완 생성은 자동 덮어쓰기 금지, `제안 -> 편집 -> 삽입` 흐름을 유지합니다.
- KOSHA/법령/재해사례/내부 지식DB를 같은 카드 UI로 묶되 출처 배지를 분리합니다.

## 추천 런칭 형태

### 지금 가능한 형태

`유료 파일럿 패키지`가 가장 적합합니다.

구성:
- 현장 한 곳 온보딩
- 대표 작업 시나리오 문서팩 생성
- 위험성평가표, 작업계획서, TBM, 안전보건교육 기록 검수
- 메일·문자 전파 설정
- 작업자 언어팩 설정
- 첫 제출용 PDF/HWPX/XLS 검토

이 방식이면 아직 셀프서브가 완전히 닫히지 않아도 고객은 돈을 낼 이유가 있습니다. “소프트웨어 구독”보다 “현장 서식 자동화 + 운영 세팅”에 가깝게 팔아야 합니다.

### 아직 피해야 하는 형태

- 카드 결제 후 즉시 모든 현장이 혼자 쓰는 셀프서브
- 공공기관 제출 자동 통과를 보장하는 표현
- 법적 효력이나 인증 대체로 보이는 표현
- 카카오·밴드까지 완전 자동 전파된다고 보이는 표현

## 첫 번째 그릴 질문

유료상품 런칭의 첫 고객을 누구로 고정할까요?

추천 답: `중소 건설·물류 현장의 안전관리자 1명 + 외국인 또는 신규 작업자가 있는 현장`으로 고정합니다.

이유:
- 한 줄 입력과 문서팩 생성 가치가 바로 보입니다.
- 위험성평가, TBM, 안전보건교육, 외국인 공지가 모두 자연스럽게 필요합니다.
- 대기업 본사 EHS보다 의사결정이 빠릅니다.
- 공공기관 제출 수준의 서식을 “보조 초안”으로 설명하기 좋습니다.

다음 질문은 이 타깃을 기준으로 `첫 유료 파일럿에서 반드시 닫아야 하는 기능 범위`입니다.
