## 2. TBM 필드/표 구조 추출 요약

필수 렌더링 섹션:

- 기본정보: 회사/현장/공사/장소/일자/시작·종료시간/진행자/기상
- 금일 작업: 작업내용, 작업순서, 장비·자재, 관련 허가서
- 위험성평가 반영: 위험요인, 원인, 감소대책, 작업중지 기준, 담당자, 복창/확인
- 근로자 확인: 보호구, 건강상태, 음주·피로, 신규·변경·외국인 근로자 이해확인
- 참석자 명단: 소속, 직종, 성명, 오전/오후, 서명, 이해확인
- 사진·증빙: TBM 실시 사진, 작업 전 상태, 통제구역, 모바일/작업일지 기록, 보관 위치
- 교육시간 인정 검토: 교육구분, 시간, 내용, 대상, 확인증빙, 조건부 검토 문구
- 미조치/후속조치 및 최종 확인: 잔여위험, 조치담당, 기한, 확인자 서명

서명·확인란: 참석자 서명, 진행자/관리감독자 확인, 현장소장 또는 확인자 서명, 복창/이해확인 체크가 필요합니다.

사진·증빙란: TBM 실시 사진, 작업 전 현장 상태, 보호구 착용, 위험구역 통제, 사진/영상 파일명, 모바일 로그 또는 작업일지 보관 위치가 필요합니다.

교육시간 인정 문구: TBM 자체를 법정교육 대체로 단정하지 말고, 위험성평가 결과와 교육 내용·대상·시간·확인증빙이 함께 남을 때 증빙 활용 가능성을 검토한다는 조건부 문구가 안전합니다.

## 4. 현재 SafeClaw TBM 렌더러와의 갭

- `components/WorkpackEditor.tsx:673-731`: TBM 일지 기본정보, 근로자 확인, 작업사항/위험요인, 교육, 참석자, 사진·영상 증빙 틀은 이미 있습니다. 다만 업체/공사명, 시작·종료시간, 위험별 조치담당/작업중지 기준, 외국인·신규자 이해확인, 교육시간 인정 판단이 구조화되지 않았습니다.
- `components/WorkpackEditor.tsx:742-780`: 브리핑은 위험성평가와 기상 신호를 TBM 전달 표로 바꾸지만, 보관용 회의록/일지의 참석자 서명표와 후속조치까지 같은 schema로 연결되지는 않습니다.
- `lib/mock-data.ts:652-690`: mock TBM 기록은 실무 문구를 포함하지만 HWPX 표 필드와 렌더러가 같은 canonical payload를 쓰지는 않습니다.
- `lib/mock-data.ts:697-731`: 교육 기록에는 TBM 교육시간 인정 검토 문구가 있으나, 렌더링 TBM 일지에는 별도 교육 인정 필드가 없습니다.
- `lib/search.ts:414-415`: 현재 runSearch는 긴 근거/사례/공식자료 appendices를 TBM 본문에 직접 붙입니다. TBM 문서는 짧은 반영 근거만 남기고 원문성 근거는 evidence UI/metadata에 분리하는 편이 맞습니다.

## 5. 최종 TBM Schema 제안

JSON 상세 schema는 동봉된 `tbm-reference-ingestion-report.json`의 `proposedTbmSchema`에 포함했습니다. 핵심은 `session`, `workSummary`, `riskBriefing`, `workerChecks`, `attendance`, `evidence`, `educationRecognition`, `signOff`를 분리하고, `documentKind`로 브리핑/일지/회의록을 구분하는 구조입니다.

## 6. Supabase 적재 후보

실제 DB mutation은 하지 않았습니다. 후보 payload와 SQL 초안은 `tbm-reference-ingestion-report.json`의 `supabasePayloadCandidate`, `sqlCandidate`에만 기록했습니다.

## 7. 이식 우선순위

- TBM canonical payload 타입부터 추가: HTML/HWPX/Excel/PDF가 같은 데이터를 쓰게 해야 표 누락을 막을 수 있습니다.
- `tbmBriefing`과 `tbmLogDraft` 분리 유지: 브리핑은 전달용, 로그/회의록은 보관·서명·증빙용으로 목적이 다릅니다.
- 근거 출력 축소: 문서 본문에는 “반영 근거: 위험성평가 결과, 기상 신호, KOSHA/TBM 가이드 확인” 수준만 두고, 긴 원문 근거는 evidence metadata로 이동해야 합니다.
- 교육 인정은 조건부 필드화: 법정교육 대체 단정 대신 시간, 대상, 내용, 서명, 사진/모바일 기록이 있을 때 검토 가능하다는 구조가 필요합니다.
