# SafeGuard 제출 직전 실행 모드

## 기준

지금부터는 기능 확장보다 제출 성공을 우선한다. 모든 작업은 아래 세 질문에 답해야 한다.

1. 접수 양식에 바로 쓸 수 있는가.
2. 발표 또는 시연에서 바로 보여줄 수 있는가.
3. 실패하면 제출을 막는가, 아니면 제출 후 보완으로 보낼 수 있는가.

## 제출용 핵심 문장

SafeGuard는 현장 작업 한 줄을 공공데이터와 AI로 분석해 위험성평가, TBM, 안전보건교육, 외국인 근로자 안내문, 현장 전파 메시지까지 연결하는 산업안전 문서팩 서비스다.

## 기술 구성 한 장 설명

| 구성 | 역할 | 제출 설명 |
| --- | --- | --- |
| Vercel/Next.js | 웹앱, 문서팩 생성, UI | 사용자가 작업 내용을 입력하고 결과를 확인하는 제품 화면 |
| Gemini | 위험요약, 문서 문장화, 다국어 안전공지 | 공공데이터 근거를 문서와 쉬운 안내문으로 변환 |
| Law.go/법제처 | 법령·해석례·판례 근거 | 위험성평가, TBM, 교육기록의 법령 근거 |
| 기상청 | 현재·예보·특보 위험 신호 | 강풍, 강수, 폭염 등 작업중지 기준 |
| KOSHA | 안전자료, 재해사례, MSDS, 가이드 | 공식자료 기반 교육·예방 근거 |
| 고용24/Work24 | 후속 교육 추천 | 사업주훈련·외국인 교육 연계 |
| Supabase | 관리자 계정, 문서팩·이력 저장 | 제출 시 운영화 축으로 설명 |
| n8n | 메일·문자 전파 자동화 | 생성된 문서팩과 현장 공지를 채널별로 전송하고 결과를 회신 |

## 제출 전 우선순위

### P0: 접수 제출에 필요한 것

- 접수 양식 입력값 확정.
- 서비스 설명 1000 Byte 이내 문장 확정.
- AI 활용명과 AI 활용 방식 확정.
- 데이터명, 기관명, 컬럼명 입력값 확정.
- Vercel Production URL이 열리는지 확인.
- 대표 시나리오 1건에서 문서팩 생성이 끝까지 되는지 확인.

### P1: 시연 안정성에 필요한 것

- `/demo`, `/why`, `/preview`, `/knowledge` 동작 확인.
- 대표 시나리오로 PDF/HWPX/XLS 다운로드 확인.
- 메일·문자 전파는 실제 발송 또는 접수 로그 중 하나를 증빙.
- 카카오·밴드는 준비 중으로 잠그고 메인 성공 기준에서 제외.

### P2: 제출 후 보완 가능

- Supabase 이력 저장 고도화.
- n8n 공개 relay 안정화.
- Google Sheets OAuth 자동 생성.
- Obsidian 검수 편의 도구.
- 추가 크롤링/seed DB 확장.

## n8n 설명 문구

> 문서팩 생성 이후에는 Oracle 서버의 n8n 워크플로우를 통해 메일·문자 전파를 자동화합니다. 웹앱은 문서 생성과 전파 요청을 담당하고, n8n은 채널별 provider 연동, 발송 결과 회신, 향후 카카오·밴드 확장 지점을 담당합니다.

## 제출 당일 금지

- 새 기능을 크게 추가하지 않는다.
- DB migration은 제출 직전 새로 실행하지 않는다.
- 실제 발송 채널을 무리하게 늘리지 않는다.
- mock/fallback 같은 내부 용어를 접수 문장에 쓰지 않는다.
- “법적 효력 보장”처럼 과한 표현을 쓰지 않는다.

## 제출 당일 확인 명령

```powershell
npm.cmd run typecheck
npm.cmd run build
npm.cmd run smoke:submission
```

최종 증거 파일은 `evaluation/submission-readiness/` 아래 결과를 사용한다.
