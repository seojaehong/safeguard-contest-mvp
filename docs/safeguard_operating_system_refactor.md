# SafeGuard Operating System Refactor

## 목적

SafeGuard의 다음 방향은 한 번 감탄하는 생성형 데모가 아니라 매일 켜는 현장 안전관리 운영 시스템입니다. 핵심 루프는 `오늘 작업 -> 문서팩 -> 근거 -> 작업자/교육 -> 전파 -> 이력 -> 감사 대응 기록`으로 고정합니다.

이번 문서는 구현 전에 제품과 기술 방향을 결정하기 위한 기준서입니다. 외부 API 호출부와 응답 파싱 규칙은 변경하지 않으며, 모든 확장은 API 응답을 받은 이후의 저장, 스냅샷, 편집, 증빙 레이어에서만 진행합니다.

## 제품 원칙

1. 재방문 우선: 사용자가 다음 날 같은 현장을 다시 열 이유를 만든다.
2. 산출물 위계화: 위험성평가표, TBM, 외국인 전송본을 1급 산출물로 두고 나머지 8종은 보조 산출물로 접는다.
3. 단일 진실 원천: 현장명, 작업자 수, 날씨, 위험요인, 통제대책은 SharedContext에서 한 번만 수정한다.
4. 책임 완화: 타임스탬프, 스냅샷, 수정 이력, 굵은 면책 고지를 통해 공식 근거 기반 보조자료임을 분명히 한다.
5. 가격 사다리: 같은 제품 위에 저장, 증빙, 승인, 다현장 운영 가치를 단계적으로 보여준다.

## 정보 구조

홈은 단일 작업 입력 화면에서 운영 대시보드로 바뀝니다. 첫 화면은 오늘 작성할 문서가 아니라 관리해야 할 현장과 최근 작업 상태를 보여줍니다.

현장 워크스페이스는 다음 순서로 고정합니다.

1. 오늘의 안전: 위험성평가표, TBM, 외국인 전송본.
2. 보조 산출물: 작업계획서, 교육기록, 비상대응, 사진/증빙, 공유 메시지 등.
3. 증빙 라이브러리: KOSHA 자료, 재해사례, 사진, 교육 확인, 전파 결과.
4. 이력: DailyEntry, AuditTrail, 저장/전파 로그.

## 산출물 위계

1급 산출물은 사용자가 가장 먼저 검토하고 현장에서 바로 실행하는 문서입니다.

| 산출물 | 역할 | 사용자 행동 |
|---|---|---|
| 위험성평가표 | 오늘 작업의 핵심 위험과 감소대책 | 확인, 보완, 출력 |
| TBM | 작업 전 회의에서 읽고 확인할 브리핑 | 참석자 확인, 공유 |
| 외국인 전송본 | 언어별 핵심 안전공지 | 언어 선택, 전송 |

보조 산출물 8종은 기본적으로 접힘 상태로 두고, 필요한 경우 열어 편집하거나 다운로드합니다. JSON, CSV, HTML, JPG 등 보조 형식은 `고급 내보내기` 안으로 이동합니다.

## 안전 문구 기준

모든 화면과 출력물은 다음 표현을 사용합니다.

- 사용: `공식 근거 기반 보조자료`, `현장 확인 필요`, `감사 대응 기록`, `관리자 검토 후 사용`.
- 금지: 최종 법률 판단처럼 보이는 표현, 공식 효력을 확정하는 듯한 표현, 현장 확인을 생략해도 된다는 표현.

표준 면책 고지는 모든 출력물 푸터에 들어갑니다.

> 본 문서는 공공데이터와 입력 정보를 기반으로 작성된 현장 검토용 초안입니다. 법령 원문, 사업장 세부 조건, 현장 실측, 관리감독자 확인을 거쳐 사용해야 합니다.

## 기존 자산 재사용

현재 구현된 API 조합, 11종 문서팩, 작업자/교육 패널, n8n 전파, Supabase 최소 저장 구조는 유지합니다. 새 리팩토링은 다음을 새로 얹습니다.

- Site 중심 재방문 구조.
- DailyEntry 기반 일자별 스냅샷.
- SharedContext 기반 공통 필드 편집.
- EvidenceLibrary 기반 증빙 관리.
- AuditTrail 기반 수정 이력.
- 플랜별 잠금 표시와 업그레이드 카피.

## 범위 밖

- 기상청, Law.go, Work24, KOSHA API 호출 함수 변경.
- 외부 API 응답 파싱 규칙 변경.
- 결제 연동.
- DB migration 즉시 실행.
- 공식 효력을 확정하는 듯한 문구 사용.
