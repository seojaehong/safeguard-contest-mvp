# SafeGuard 코드리뷰 및 현재 루브릭 검토

## 검토 기준

- 대상 브랜치: `codex/safety-evidence-foreign-services`
- 비교 기준: `master...HEAD`
- 검증 명령: `npm.cmd run typecheck`, `npm.cmd run build`
- 검토 축: 현장 전파, 저장 이력, 법령 근거, 외국인 메시지, 수상·유료화 루브릭

## 코드리뷰 Findings

### 중요도 보통: n8n 이메일 수신자 판별 정규식이 정상 이메일을 누락할 수 있음

- 파일: `docs/n8n_safeguard_workflow_template.json`
- 위치: embedded n8n code node, `recipientEmails()`
- 내용: 이메일 판별식이 `^[^s@]+@[^s@]+.[^s@]+$` 형태라 `s` 문자가 들어간 정상 이메일을 제외합니다. 기본 근로자 예시의 `supervisor@safeguard.local` 같은 주소는 수신자에서 빠지고, 실제 발송은 기본 SMTP 계정 또는 기본 수신자로 fallback될 수 있습니다.
- 영향: 현장 전파에서 선택한 이메일 수신자와 실제 발송 대상이 달라질 수 있습니다.
- 권장 조치: 정규식을 `^[^\s@]+@[^\s@]+\.[^\s@]+$`로 교체하고, n8n 운영 workflow에도 같은 수정본을 import합니다.

### 중요도 보통: 교육 이력 저장 실패를 부분 성공으로 명확히 분기하지 않음

- 파일: `components/FieldOperationsWorkspace.tsx`
- 위치: `saveWorkspace()`
- 내용: `educationResponse.ok`를 확인하지 않고 `onSavedWorkpack()`과 성공 메시지 조합을 진행합니다. 교육 기록 API가 실패해도 문서팩 저장 ID가 전파 패널에 연결되어, 사용자는 교육 이력까지 저장됐다고 오해할 수 있습니다.
- 영향: 유료화 관점의 핵심인 교육 확인 이력과 문서팩 이력이 불일치할 수 있습니다.
- 권장 조치: 교육 이력 실패 시 `partial` 상태로 표시하고, 전파 로그 저장 전에 교육기록 미저장 상태를 노출합니다.

## 결론

현재 상태는 제출과 시연 기준으로는 강한 편이며, 수상권 진입을 노릴 수 있는 완성도입니다. 최우수권 안정화를 위해서는 기능을 더 늘리기보다 전파 정확도, 교육 이력 정합성, 근거 반영 위치를 더 명확히 만드는 것이 우선입니다.

유료화는 셀프서브 SaaS보다 파일럿/현장 커스터마이징 결합형이 현실적입니다. 다음 작업은 `n8n 이메일 정규식 수정`, `교육 이력 partial 상태 처리`, `근거-문서 반영 문장 UI 고정` 순서가 적절합니다.
