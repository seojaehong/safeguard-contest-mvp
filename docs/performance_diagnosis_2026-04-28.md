# SafeGuard 성능 진단 및 개선 계획

## 현재 증상

프로덕션 측정 기준 홈 `/`과 `/api/ask`가 느립니다.

- `/`: 약 29초대
- `/api/ask`: 약 37초대
- `/search`: 약 3초대

따라서 병목은 CSS나 카드 렌더링보다 `문서팩 생성 경로`에 있습니다.

## 주요 원인

### 1. 홈 첫 렌더링이 `runAsk()` 완료를 기다림

`app/page.tsx`에서 홈 서버 렌더링 중 `await runAsk(q)`를 바로 수행합니다. 이 때문에 사용자는 첫 화면을 보기 전에 법령, 기상, 교육, KOSHA, 재해사례, Gemini 생성이 모두 끝날 때까지 기다립니다.

### 2. 공공데이터 enrichment 후 Gemini 생성을 시작하던 구조

기존 `runAsk()`는 Law.go, 기상청, Work24, KOSHA 교육, KOSHA 공식자료 호출을 먼저 기다린 뒤 Gemini 문서 생성을 시작했습니다. Gemini 생성은 citations만 있으면 시작할 수 있으므로, 나머지 enrichment와 직렬로 기다릴 필요가 없습니다.

### 3. 외부 API timeout 정책이 데모 안정성 중심

공공 API는 timeout 20초, retry 1회가 기본입니다. 안정성에는 좋지만 홈 첫 화면에 직접 걸리면 느린 API 하나가 전체 체감 속도를 끌어내립니다.

### 4. 홈이 결과 화면과 작업 입력 화면을 동시에 담당

제품 UX상 첫 화면은 빠르게 떠야 하는데, 현재는 입력 전/후 결과 화면이 같은 서버 렌더링 경로에 있어 항상 무거운 작업을 수행합니다.

## 이번 반영

- `runAsk()`에서 citations 확보 후 Gemini 생성을 먼저 시작하고, 기상·교육·KOSHA·재해사례 enrichment와 병렬로 기다리도록 변경했습니다.
- 기능 의미는 유지하면서 직렬 대기 시간을 줄이는 1차 개선입니다.

## 다음 개선 우선순위

1. 홈 `/`은 즉시 뜨는 작업공간 shell로 만들고, 문서팩 생성은 client-side `/api/ask` 호출로 분리합니다.
2. 기본 대표 시나리오는 최근 smoke JSON 또는 캐시된 결과를 먼저 보여주고, 최신 live 결과는 백그라운드 갱신합니다.
3. non-critical enrichment는 단계적으로 표시합니다. 예: 먼저 위험 요약과 문서팩 초안, 이후 KOSHA 교육/재해사례/후속 교육 카드 보강.
4. API별 latency logging을 `evaluation/performance/`에 남기고, 5초 이상 지연된 source를 상태 패널에 표시합니다.
5. Vercel Edge Config, KV, Supabase cache 또는 파일 기반 snapshot으로 대표 시나리오 응답을 캐시합니다.

## 판단

수상/시연 기준에서는 홈 첫 화면이 느린 것이 가장 위험합니다. 제출 전 최우선 성능 전략은 “첫 화면 즉시 표시, 생성은 진행 상태로 보여주기”입니다. 지금 반영한 병렬화는 1차 응급처치이고, 체감 UX를 확실히 바꾸려면 홈 렌더링과 문서팩 생성 작업을 분리해야 합니다.
